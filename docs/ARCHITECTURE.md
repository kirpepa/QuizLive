# Архитектура

## Обзор

```
┌───────────────────────────┐        REST (JWT)          ┌────────────────────────────┐
│   Клиент (React + Vite)   │  ───────────────────────►  │      Express API           │
│                           │                            │  auth / quizzes / sessions  │
│  - Кабинеты, редактор     │  ◄───────────────────────  │  / me   + статика /uploads  │
│  - Экраны ведущего/игрока  │                            └────────────┬───────────────┘
│                           │                                         │ Prisma
│          Socket.IO client │  ═══════ WebSocket ══════►  ┌───────────▼────────────────┐
└───────────────────────────┘   room:*, question:*, quiz:*│   Socket.IO + sessionMgr   │
                                                          │  (живое состояние комнат)   │
                                                          └───────────┬────────────────┘
                                                                      │
                                                              ┌───────▼────────┐
                                                              │ SQLite (Prisma) │
                                                              └────────────────┘
```

Два канала связи:

- **REST** — всё, что не требует реального времени: регистрация/логин/refresh,
  CRUD квизов и вопросов, загрузка изображений, создание сессии, история
  кабинетов.
- **WebSocket (Socket.IO)** — проведение сессии: подключение к комнате,
  трансляция вопросов, приём ответов, показ ответа (reveal), лидерборд, финал.

## Ключевое решение: серверный таймер — источник истины

Состояние идущей сессии живёт **в памяти сервера** (`sessionManager.js`,
`Map<roomCode, room>`). Именно сервер:

- хранит дедлайн текущего вопроса (`questionEndsAt`) и запускает `setTimeout`;
- отклоняет ответы, пришедшие после дедлайна (`Date.now() > questionEndsAt`) или
  вне фазы показа вопроса;
- начисляет баллы, зная точное оставшееся время;
- инициирует reveal по таймауту **или** когда ответили все подключённые (если
  правило «смена ответа» выключено).

Клиентский таймер (`Timer.jsx`) — только визуализация: он синхронизируется с
серверными `endsAt`/`serverNow`, компенсируя расхождение часов, но не влияет на
приём ответов. Так участник не может «дорисовать» себе время.

Результаты фиксируются в БД в момент reveal (`ParticipantAnswer` + обновление
`SessionParticipant.score`), поэтому история сохраняется, даже если комната позже
выгрузится из памяти.

## Жизненный цикл сессии

```
pending ──quiz:start──► question ──(таймаут | все ответили | quiz:reveal)──► reveal
   ▲                        ▲                                                   │
   │                        └──────────────── quiz:next ─────────────────────┘
 создание                                          │
 (REST POST /sessions)                     последний вопрос → quiz:finish → finished
```

Статусы в БД (`pending/active/finished`) и в памяти комнаты
(`lobby/question/reveal/finished`) — разные словари для разных задач и не
конфликтуют.

## Протокол WebSocket

**Организатор → сервер:** `organizer:open`, `quiz:start`, `quiz:reveal`,
`quiz:next` (все с ack-колбэком для ошибок).

**Участник → сервер:** `room:join` (с необязательными `participantId` +
`rejoinToken` для переподключения), `question:answer`.

**Сервер → комната/клиенту:**

| Событие                   | Назначение                                     |
|---------------------------|------------------------------------------------|
| `room:participants`       | обновлённый список участников                  |
| `question:show`           | вопрос без флага правильного ответа + дедлайн   |
| `question:progress`       | ведущему: сколько ответили                      |
| `question:answered_ack`   | участнику: ответ принят                         |
| `question:result`         | участнику: верно/неверно + баллы                |
| `question:reveal`         | правильные варианты + промежуточный лидерборд   |
| `quiz:finish`             | финальный лидерборд                             |

Клиентам **никогда** не отправляется флаг `isCorrect` до reveal — `question:show`
отдаёт варианты без него; правильные ответы приходят только в `question:reveal`.

## Личность участника и переподключение

При первом `room:join` сервер создаёт участника и возвращает пару
`{ participantId, rejoinToken }`; клиент хранит их в `localStorage`. При обрыве и
повторном `connect` клиент снова шлёт `room:join` с этой парой.

- Реконнект к существующему участнику разрешён, только если совпал
  **`rejoinToken`** (секрет, выданный при первом входе) **или** `userId` из JWT.
  Знание одного лишь `participantId` (он рассылается в списке участников) для
  захвата недостаточно — защита от подмены игрока.
- Авторизованный участник дедуплицируется по `(sessionId, userId)`: даже если он
  потерял `participantId`, повторный вход переиспользует его запись, а не плодит
  дубликаты в лидерборде и истории.

Если участник или организатор переподключается **в фазе показа ответа**, сервер
досылает закешированный `lastReveal` (правильные варианты + лидерборд), так что
экран не остаётся пустым.

## Безопасность

- Пароли — bcrypt; доступ к приватным роутам — JWT-middleware; операции с квизами
  проверяют владельца (`loadOwnedQuiz`), а организаторские socket-действия — роль
  и владение сессией.
- Ответы валидируются на сервере (тип вопроса, число вариантов, дедлайн).
- Загрузка файлов ограничена типом и размером (5 МБ).
